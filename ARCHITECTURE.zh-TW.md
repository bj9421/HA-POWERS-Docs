## 🧠 設計理念 — 為什麼要有這個

> **核心問題：** 既然可以直接「開始寫程式」，為什麼要搞一套 7 階段、帶門控的管線？

### 我們要解決的問題

沒有結構化的開發管線，AI 輔助開發有三個慢性毛病：

| 失敗模式 | 發生什麼事 | HA-POWERS 怎麼修 |
|-------------|-------------|----------------------|
| **做錯東西** | 直接跳去寫程式 → 後來發現規格誤解 → 全部重寫 | 第 1 階段（腦力激盪）強制先寫規格書，用戶批准後才開始寫程式 |
| **進度不可預測** | 「只是個小功能而已」→ 三天後 → 什麼都沒做完 | 第 2 階段（撰寫計畫）把一切拆解成 2–5 分鐘的小任務，附精確檔案路徑與命令 |
| **沒有負責任機制** | 變更在真空中發生 → 沒有審查 → bug 上線 | 第 5 階段（程式碼審查）+ 第 6 階段（PR + CI）在合併前建立強制品質門檻 |

### 為什麼是 7 個階段？

每個階段解決一個特定的失敗模式。它們是**順序且受門控的**——當前階段產出工件之前，不能進入下一個：

| 階段 | 解決 | 門控輸出 |
|-------|--------|-------------|
| 1. 腦力激盪 | 「我們做錯了東西」 | 已核准的設計規格 |
| 2. 撰寫計畫 | 「我不知道從哪裡開始」 | 附精確檔案路徑的任務清單 |
| 3. Git Worktrees | 「我的 main 分支太亂了」 | 隔離的工作區 |
| 4. 子代理開發 | 「不用 TDD 寫程式就像賭博」 | 功能程式碼 + 通過的測試 |
| 5. 程式碼審查 | 「我漏掉了自己程式碼的 bug」 | 人工級別的審查 |
| 6. PR + CI | 「變更直接推到 main」 | 已合併的 PR + 綠色 CI |
| 7. 清理 | 「Git 歷史充滿被遺棄的分支」 | 乾淨的 main 簽出 |

**門控是關鍵。** 每個階段必須在下一個開始前產出具體的工件。這防止了常見的壞習慣——「先開始寫再說」——這種想法創造了最昂貴的返工。

### HA-POWERS 在 Superpowers 之上加了什麼

[obra's Superpowers](https://github.com/obra/superpowers) 開創了漸進式揭露的技能模式。HA-POWERS 在其基礎上增加了：

- **協調層** — Superpowers 提供個別技能。HA-POWERS 提供帶門控的**管線**，將它們串接起來。
- **進度追蹤器** — 一個可見的核對清單，讓你清楚知道自己在哪個階段。
- **階段門控** — 明確的條件，必須滿足才能繼續前進。
- **單一設定檔架構** — 一個 Hermes 設定檔處理所有 7 個階段。只有第 4 階段可能產生 0–2 個暫時性子代理。
- **看板整合** — 可選的視覺化面板，用於多功能追蹤。
- **決策樹** — 自動偵測何時該跑完整管線、何時該跳過直接快速修復。

### 指導原則

> **每個功能，每次都一樣，從構想到合併 PR，一步不漏。**

這不是官僚——這是**防止未來的自己忘記今天為什麼做這個決定的保險**。規格書、計畫、PR 描述——這些都是你（或你的審查者）之後會感謝自己的工件。

## 🚧 進度追蹤器

在每個功能的**開始**展示這個。每完成一個階段就更新核取方塊。

### 第 1 階段：腦力激盪
- [ ] 探索上下文與程式碼庫
- [ ] 提出釐清問題
- [ ] 提出 2-3 種方案
- [ ] 展示設計與架構
- [ ] 將規格寫入 `<project>/docs/specs/`
- [ ] 自我審查規格（不留佔位符）
- [ ] 用戶核准規格 ✅

### 第 2 階段：撰寫計畫
- [ ] 閱讀已核准的規格
- [ ] 探索程式碼庫模式
- [ ] 撰寫任務（每個 2-5 分鐘）
- [ ] 包含精確檔案路徑
- [ ] 每個任務都包含 TDD 循環
- [ ] 審查計畫完整性
- [ ] 儲存至 `<project>/docs/plans/` ✅

### 第 3 階段：Git Worktrees
- [ ] 偵測現有的 worktree
- [ ] 在 feat/<name> 建立 worktree
- [ ] 安裝相依套件
- [ ] 驗證測試基準乾淨
- [ ] 隔離工作區就緒 ✅

### 第 4 階段：子代理開發
- [ ] 閱讀計畫，提取任務
- [ ] 決定子代理數量
- [ ] 派遣開發者子代理
- [ ] 派遣審查者子代理（大型功能）
- [ ] 修復問題 → 重新審查循環
- [ ] 所有任務完成
- [ ] 最終整合檢查 ✅

### 第 5 階段：程式碼審查
- [ ] 審查正確性
- [ ] 審查可維護性
- [ ] 審查安全性
- [ ] 審查效能
- [ ] 審查測試覆蓋率
- [ ] 報告發現（嚴重程度）
- [ ] 修復問題 → 重新審查 ✅

### 第 6 階段：PR + CI
- [ ] 推送分支到 GitHub
- [ ] 建立附描述的 PR
- [ ] 監控 CI 狀態
- [ ] 自動修復 CI 失敗（最多 3 次）
- [ ] 合併（squash + 刪除分支）✅

### 第 7 階段：清理
- [ ] 移除本地 worktree
- [ ] 切回 main
- [ ] 清理分支
- [ ] 功能交付 ✅

## 📊 階段門控

| 階段 | 門控條件 | 輸出工件 |
|-------|---------------|-----------------|
| 1. 腦力激盪 | 用戶核准書面規格 | `<project>/docs/specs/<日期>-<主題>-設計.md` |
| 2. 撰寫計畫 | 計畫已儲存並提交 | `<project>/docs/plans/<日期>-<主題>-計畫.md` |
| 3. Git Worktrees | Worktree 建立完成，測試通過 | 隔離的 `./worktrees/feat/<名稱>` |
| 4. 子代理開發 | 所有任務完成，測試通過 | 分支上的功能程式碼 |
| 5. 程式碼審查 | 審查通過 | 已審查的程式碼 |
| 6. PR + CI | PR 已合併 | 關閉的 PR + 已刪除的分支 |
| 7. 清理 | Worktree 已移除 | 乾淨的 main 簽出 |

## 🎯 架構哲學

### 一個設定檔，不是很多個

> 🎯 **一個設定檔 = 完整管線。** 不是 6 個設定檔，不是 6 個代理。

| 術語 | 定義 | 生命週期 | 獨立性 |
|------|-----------|----------|-------------|
| **設定檔** | 完整的 Hermes 配置 | 永久 | 有自己的身分 |
| **子代理** | 暫時的子程序 | 分鐘級（一次性） | 繼承父設定檔 |
| **對話** | 你正在進行的對話 | 聊天持續期間 | 對話歷史 |

**一個 Hermes 設定檔執行整個 HA-POWERS 管線。** 所有子代理都是暫時的子程序——它們不需要自己的設定檔。

### 最少子代理

HA-POWERS 進行實際工作**最多只用 0–2 個子代理**：

```
                 ┌──────────────────────────────────┐
                 │  協調者（你 + Hermes）              │  ← 1 個設定檔
                 │  直接處理所有階段：                  │
                 │  • 第 1 階段：與用戶討論（規格）     │
                 │  • 第 2 階段：撰寫計畫               │
                 │  • 第 3 階段：git worktree           │
                 │  • 第 5 階段：lint/安全檢查          │
                 │  • 第 6 階段：git push / gh pr       │
                 │  • 第 7 階段：清理                   │
                 └──────────────┬───────────────────┘
                                │
                    只有第 4 階段：│ delegate_task
                                ▼
                 ┌──────────────────────────────────┐
                 │  1–2 個子代理（暫時性）              │
                 │  ┌────────────┐  ┌────────────┐  │
                 │  │ 開發者      │  │ 審查者      │  │
                 │  │ （寫程式）  │  │ （稽核）    │  │
                 │  └────────────┘  └────────────┘  │
                 │  • 由 delegate_task 產生           │
                 │  • 交付報告後銷毀                  │
                 │  • 與協調者使用同一設定檔          │
                 │  • 無持久狀態                      │
                 └──────────────────────────────────┘
```

| 階段 | 需要子代理？ | 為什麼 |
|-------|----------------|------|
| 1. 腦力激盪 | ❌ | 直接跟用戶討論 |
| 2. 撰寫計畫 | ❌ | 寫很快，委派很慢 |
| 3. Git Worktrees | ❌ | `git worktree add` 一行搞定 |
| **4. 開發** | **🟢 1 開發者** | 隔離的 TDD 工作負載 |
| **4. 審查** | **🟢 1 審查者（可選）** | 僅大型功能 |
| 5. 品質門控 | ❌ | CLI 命令 |
| 6. PR | ❌ | `git push && gh pr create` |
| 7. 清理 | ❌ | `git worktree remove` |

## 📋 看板整合

看板是**完全可選的**。無論有沒有它，每個階段的運作方式都一樣。

### 管線 ↔ 看板對應

| 階段 | 看板動作 | 觀察 |
|-------|--------------|-------------|
| 1. 腦力激盪 | `kanban add "功能..." --col backlog` | 用戶核准規格 |
| 2. 撰寫計畫 | `kanban move K1 --col todo` | 計畫已提交 |
| 4. 子代理開發 | `kanban move K1 --col doing` | 子代理已派遣 |
| 4. 開發者完成 | `kanban log K1 "✅ Dev complete"` | 測試通過 |
| 4. 逐任務審查 | `kanban log K1 "🔍 Spec PASS"` | 兩次審查都通過 |
| 4. 任務完成 | `kanban move K1 --col review` | 所有任務已實作 |
| 5. 程式碼審查 | `kanban move K1 --col review` → `--col done` | 審查通過 |
| 6. PR 已合併 | `kanban log K1 "🔀 PR #42 merged"` | CI 綠色，已合併 |
| 7. 清理 | `kanban archive --days 7` | 週末清理 |

### 為什麼基於檔案的方案適合邊緣裝置

- **零相依性** — 純 Python stdlib，不需要伺服器、不需要資料庫
- **Git 原生** — `KANBAN.json` 隨程式碼一起版本控制
- **離線可用** — 沒有網路也能運作（邊緣模式）
- **可程式化** — Hermes 透過終端機讀寫
- **可觀察** — `kanban list` 會印出到 Telegram
- **可整合** — `kanban board` 匯出到 Obsidian

## 🚫 常見錯誤

1. **為「簡單」功能跳過腦力激盪** — 第一大壞習慣。簡單的功能藏著最多的假設。
2. **沒讀規格就寫計畫** — 計畫必須能追溯回已核准的需求。
3. **在 worktree 裡面建 worktree** — 第 3 階段的偵測步驟可防止這個問題。
4. **跳過規格符合性審查** — TDD 抓得到 bug，但規格審查抓得到「做錯了東西」。
5. **CI 還沒綠就合併** — 一定要確認。
6. **產生太多子代理** — 你不需要分開的 Architect/Planner/DevOps 代理。越多只是越多雜訊。
7. **合併後留下 worktree** — 執行第 7 階段，不然它們會越積越多。
8. **Worktree 要求專案完全受追蹤** — `git worktree add` 只會簽出 git 歷史中的檔案。未追蹤的檔案不會被包含。修正：先 `git add` + commit，或直接跳過 worktree。

## 🎓 比較：進度追蹤器 vs 看板

| 面向 | 進度追蹤器 | 看板 |
|------|-----------|------|
| **目的** | 追蹤單一功能開發 | 追蹤多個功能/任務 |
| **粒度** | 每個階段內的步驟 | 附子任務的卡片 |
| **生命週期** | 功能完成後消失 | 跨會話持續存在 |
| **儲存** | 顯示在對話中 | `KANBAN.json` 檔案 |
| **場景** | 「我今天在做一个功能」 | 「這週我在追蹤 5 個功能」 |

**搭配使用：** 看板決定要做哪個功能 → 進度追蹤器跟著開發步驟走。

## ⚡ 重點整理

1. **每個階段都有門控** — 必須通過才能進入下一個
2. **進度追蹤器永遠可見** — 你永遠知道自己在哪裡
3. **看板持久化** — 跨會話追蹤所有功能
4. **對 Git 友善** — 所有輸出都是檔案，受版本控制
5. **零相依性** — 純 Python stdlib，不需要伺服器或資料庫
6. **彈性** — 小功能可以跳過階段（見決策樹）
